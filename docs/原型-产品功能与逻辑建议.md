# 文件管理原型 — 产品功能与逻辑建议（仅原型/产品视角）

> 只从**产品功能、用户场景、信息与交互逻辑**出发，不考虑技术实现。适用于模拟原型阶段的补全与迭代。

---

## 一、功能闭环与「可点即可信」

| 建议 | 说明 |
|------|------|
| **入口与出口一致** | 每个有文案的按钮/入口，在原型里最好都有对应的「结果」：要么进入下一屏、要么弹出浮层/提示。例如「上传」「新建」「权限」若只做 toast「演示中」，可在原型里增加**简版浮层**（上传：选文件/拖拽区 + 确定；新建：输入文件夹名 + 确定；权限：当前目录的权限列表占位），不接真实接口，但**流程能走通**，演示时更可信。 |
| **分享的反馈** | 「分享」目前是「链接已复制」。原型里可加一步：复制后浮层或 toast 写明「分享链接已复制，可发送给同事」，或弹一个简版「分享设置」浮层（仅展示：链接、有效期、权限只读/可下载），让评审/用户理解「分享」到底代表什么。 |
| **收藏可操作** | 「我的收藏」若一直为空，会显得未完成。建议：在列表/树里每个文档增加「收藏」图标（空心/实心），点击后该文档出现在「我的收藏」下，且图标变为已收藏状态；这样收藏与最近使用两条线都「有因有果」。 |

## 二、信息层次与「在哪干什么」

| 建议 | 说明 |
|------|------|
| **绩效文档首页的「待归档」可点** | 若已有「待归档」数量展示，可做成可点击入口：点击后进入一个「待归档列表」页或浮层，列出哪些部门/月份尚未归档，甚至 mock 一个「一键归档」按钮，方便讲清楚「从待办到已归档」的完整故事。 |
| **全部文件：当前所在位置** | 用户展开树并进入某一层（如 03月）后，容易不知道「当前在看哪一层」。建议：点击树节点后，**面包屑**随之变为「全部文件 > 月度绩效归档 > 2026年 > 03月」，且该层下的文件列表若单独展示在下方，则与树在视觉上有一致对应（例如左侧树高亮当前节点、右侧为当前层文件），这样「路径」和「内容」一一对应。 |
| **部门文档：和绩效的关系** | 列表里有「绩效」按钮并弹出「关联绩效任务」已很好。可再补一句产品逻辑：在绩效文档首页的部门卡片上，用一句话说明「这里展示的是各部门按月提交的绩效相关文档」，让「绩效文档」与「部门文档列表」的关系更清晰。 |

## 三、列表与预览的体验逻辑

| 建议 | 说明 |
|------|------|
| **预览时的上下文** | 预览浮层有上一份/下一份时，用户容易不知道「当前是第几个、一共几个」。建议在预览顶部或底部加一行文案：「第 2 份，共 5 份」或「2/5」，这样在原型演示时也能说清「顺序即当前列表顺序」。 |
| **从不同入口进预览的预期** | 从「部门文档列表」进预览，用户会默认上一份/下一份是在**该部门该月**内切换；从「全部文件」某月下进预览，则是在**该月文件**内切换；从「AI 搜索结果」进预览，则是在**本次搜索结果**内切换。原型里只要保证这三种情况下的顺序符合各自列表即可，必要时在预览里用一句小字提示「当前在 售前部 03月 文档中」增强可理解性。 |
| **空列表的引导** | 部门文档列表为空时，除空状态插图外，可加一句操作引导，如「该部门本月暂无文档，可前往全部文件上传或从其他月份查看」，把用户引到「有内容」的入口。 |

## 四、搜索与 AI 搜索

| 建议 | 说明 |
|------|------|
| **全部文件搜索：结果从哪来** | 搜索结果是「全部文件」下的文件，这一点在原型里可通过面包屑或标题体现，例如结果区标题写「在「全部文件」中搜索「xx」共 N 个结果」，避免用户误以为是全站或仅当前文件夹。 |
| **AI 搜索：无结果时** | 除「未找到」文案外，可给建议：「试试换个关键词」或「您也可以到全部文件中按文件夹查找」，并提供一个「去全部文件」的按钮，形成闭环。 |
| **AI 搜索：结果与预览** | AI 结果卡片已有预览/下载/分享，若点击预览，上一份/下一份建议限定在**本次 AI 返回的几条结果**内，并在预览里用一句「来自 AI 搜索」或「本批结果 2/4」之类的小字说明来源，避免和「部门列表」「树」的预览顺序混淆。 |

## 五、权限与多角色（若原型需体现）

| 建议 | 说明 |
|------|------|
| **只读/无权限的体现** | 若需要区分「有权限」和「无权限」，可在部分 mock 文档上标一个「仅查看」或小锁图标，点击预览可看，但无下载按钮或下载为置灰并提示「无下载权限」。这样评审时能讲清权限维度。 |
| **权限入口的含义** | 全部文件顶部的「权限」若点开，可用一个简版浮层说明「当前查看的是整个文件库的权限概览」或「按文件夹管理权限」，并 mock 一个权限列表（文件夹名 + 谁可看/可编辑），不必真实鉴权，但产品含义要清晰。 |

## 六、小但加分的产品细节

| 建议 | 说明 |
|------|------|
| **文档类型与预览内容一致** | 预览里已按 PDF/Word/Excel/PPT 做占位展示，保持「列表里的类型图标/后缀」与「预览内占位」一致即可，避免出现列表写 PDF、预览是 Excel 的占位。 |
| **时间与部门信息** | 在列表行或预览头部展示「提交人、时间、部门」，能让原型更贴近真实使用场景；若已有，保持统一即可。 |
| **批量操作后的反馈** | 批量下载/批量权限后，除 toast 外可加一句「已选 3 个文档，正在处理…」或「权限已更新」，关闭批量栏后列表可 mock 一个短暂「已更新」状态（如勾选消失、或某处角标变化），让操作有结果感。 |
| **底部导航的当前页** | 已在三个 Tab 间高亮当前页，若从「全部文件」进入某个树的深层再预览，底部仍保持「全部文件」选中即可，避免用户以为「丢了」入口。 |

---

## 七、建议优先补全的几项（原型演示用）

1. **收藏**：列表/树可点收藏，且「我的收藏」下能看到已收藏项，形成闭环。  
2. **面包屑**：全部文件里点击树节点后，面包屑变为完整路径，并可点击某一级回到该层。  
3. **预览上下文**：预览内增加「第 x / 共 y 份」或「当前在 [部门/月份/搜索结果] 中」。  
4. **上传/新建/权限**：用简版浮层把「点按钮 → 出现表单/列表 → 确定」走通，不接真实接口即可。  
5. **空状态与无结果**：统一一句「接下来可以做什么」的引导（如去全部文件、换关键词）。  
6. **待归档**：若展示待归档数量，则提供可点击入口和简版「待归档列表」或浮层，把绩效闭环讲清楚。
